Standard URL Syntax

The Thymeleaf Standard Dialects —Standard and SpringStandard— offer a way to easily create URLs in your web application that include URL Rewriting. This is done by means of the “at” syntax, enclosing expressions in @{...}

Context-relative URLs

The most used type of URLs are the context-relative ones. These are URLs which are supposed to be relative to the application once it is installed on the web/application server (relative to a context, therefore).

These URLs start with /:

@{/order/list}

If our app is deployed at http://localhost:8080/myapp, this URL will output as /myapp/order/list

Absolute URLs

Absolute URLs allow you to create links to other servers, including protocol (HTTP or HTTPS):

@{http://www.thymeleaf/documentation.html}

Server-relative URLs

Server-relative URLs are quite similar to context-relative URLs, except they do not assume you want your URL to be linking to a resource inside your application's context, and therefore allow you to link to a different context in the same server:

@{~/billing-app/showDetails.htm}

Your own context will be ignored, therefore even if our app is deployed at http://localhost:8080/myapp, this URL will output as /billing-app/showDetails.htm

(Note that server-relative URLs are available since Thymeleaf 2.0.5)

URL Rewriting

You might be saying: OK, this was fine, but... why did you say I needed this? What is that URL rewriting thing?

In a Java web application, URL rewriting has two functions needed for context-relative URLs:

  • Automatically detect whether the user has cookies enabled or not, and add the ;jsessionid=... fragment to the URL if not --or if it is the first request and cookie configuration is still unknown.
  • Automatically add the context name to context-relative URLs.

These two features are not applied in URLs of types other than context-relative.

Adding parameters

Many URLs in our applications have parameters... how do we add them to @{...} syntax expressions? Simple:

@{/order/details(id=3)}

You can add several, separating them with commas:

@{/order/details(id=3,action='show_all')}

Using expressions inside URLs

You might have noted that in the previous URLs...

@{/order/details(id=3)} @{/order/details(id=3,action='show_all')}

Both 3 and 'show_all' are specified as literals —a number and a text literal, respectively—. But... can we use other expressions like variable evaluations, i18n, conditionals...? Yes, we can:

@{/order/details(id=${order.id})} @{/order/details(id=${order.id},action=(${user.admin} ? 'show_all' : 'show_public'))}

What's more: an URL like:

@{/order/details(id=${order.id})}

...is in fact a shortcut for:

@{'/order/details'(id=${order.id})}

...which means that the URL base itself can be specified with an expression:

@{${detailsURL}(id=${order.id})} @{#{orders.details.localized_url}(id=${order.id})}

...even complex expressions can be used, including conditionals, for example:

@{(${user.admin}? '/admin/home' : ${user.homeUrl})(id=${order.id})}
